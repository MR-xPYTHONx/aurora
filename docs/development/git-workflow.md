# Git Workflow

## Branch naming
- Use short, descriptive branch names in lowercase.
- Prefer formats such as `feature/short-description`, `fix/issue-name`, or `chore/task-name`.
- Avoid personal names and vague labels.

## Commit message format
- Use a clear summary in the present tense.
- Follow a simple structure: `type: summary`.
- Examples: `feat: add user search`, `fix: resolve login redirect`, `chore: update dependencies`.

## Pull request checklist
- Ensure the change is scoped and tested.
- Include a brief summary of what changed and why.
- Confirm documentation or screenshots are updated when needed.
- Make sure the PR is ready for review and does not include unrelated changes.

## Code review rules
- Review for correctness, clarity, and maintainability.
- Keep feedback constructive and specific.
- Ask questions when behavior is unclear instead of assuming intent.
- Approve only when the change is safe, complete, and consistent with project standards.
